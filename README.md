# Performance Arena — Healthcare BPO Prototype

## What this is

**Performance Arena** is a clickable, gamified Performance Management prototype for a healthcare BPO operation. It demonstrates how a single platform can serve three audiences from one shared dataset:

- **Agents** — daily performance scorecard, missions, peer challenges, leaderboards, an arena store, broadcasts/training/PKTs
- **Team Leads** — coach console, team commercial exposure, risk-agent triage, mission/broadcast/training/PKT authoring, reward approvals, coaching and recognition
- **Managers** — account-level command center, SLA health, commercial impact, what-if simulator, team comparison, adoption metrics

It is a static single-page app (HTML + CSS + vanilla JS) backed by a synthetic dataset. No backend, no auth, no install — the whole thing runs from a folder.

---

## How to run it on desktop

1. Double-click **`launch.bat`**. This starts a local Python static server on port **5173** and opens [http://localhost:5173](http://localhost:5173) in your default browser.
2. The Arena Home (Agent role) loads automatically. If the page is blank, Ctrl+Shift+R to force-refresh.
3. To stop, close the launched terminal window (Ctrl+C also works).

The only requirement is **Python 3** on PATH (used purely for `python -m http.server`). Nothing to install, no `npm install`, no build step.

If you'd rather start it manually:

```cmd
python -m http.server 5173
```

The launcher prints a `http://<your-LAN-IP>:5173/` URL alongside the localhost one. On a phone or tablet on the **same Wi-Fi**, paste that URL into the browser. Layouts are responsive:

## Role switching

There is no login screen — role and active user are toggled in-app:

- **Desktop:** sidebar → click the small role badge under the avatar to switch role; the user dropdown next to it picks the active person.
- **Mobile:** tap the hamburger (top-left) → role chips at the top of the menu, user picker below.

The page resets to that role's home view on switch.

## Resetting prototype state

The app caches simulation state (challenges accepted, rewards redeemed, training completed, etc.) in `localStorage` under the key `arena_state_v7`. To go back to a clean seed:

- **In-app:** Agent role → Profile → "Reset prototype state" → confirm.
- **DevTools:** open the console and run

  ```js
  localStorage.removeItem('arena_state_v7'); location.reload();
  ```

After reset, the app reboots to Agent / `AG001` / Arena Home.

## Known limitations

This is a prototype for demos. The boundaries are intentional:

- **No backend.** Everything is local — there is no API, no database, no auth.
- **Single-browser state.** Mutations (redemptions, completions, recognitions) are stored in `localStorage` for that one browser. Two browsers see independent states.
- **Role switching is a UI affordance, not authorization.** Anyone with the link sees every role.
- **Synthetic data.** All names, KPIs, commercial figures, and recognition stories are generated by `generate_mock_data.py`. None of it represents a real client, agent, or operation.
- **Healthcare KPIs are illustrative.** SLA targets, penalty/reward slabs, and what-if curves are tuned for storytelling, not contractual accuracy.
- **Currency is USD throughout.** Numbers display as `$`/`$K`/`$M`.
- **Mutations are non-persistent across regenerations.** If you re-run `python export_to_json.py`, it overwrites `data.js` from the workbook, but you'll still see your in-browser local mutations until you reset state.
- **Tested on Chromium browsers** (Chrome, Edge). Firefox should work; Safari is untested.
- **localStorage size.** The seeded bundle is ~4.5 MB. Don't substantially expand entity counts in `generate_mock_data.py` without checking the per-origin storage quota.
- **Tailwind CDN warning.** DevTools shows a "cdn.tailwindcss.com should not be used in production" notice — this is informational, not an error. The CDN is a deliberate prototype choice to keep the project build-free.

## Client Outcome Layer

This package includes a demo-facing Client Outcome Layer:

- Agent views focus on member-impact behaviors, not dollar savings.
- Team Lead views show coaching metrics such as Repeat Contact Risk, Member Effort Risk and Access Friction with question-mark definitions.
- Manager views show modeled client-value indicators such as Repeat Contact Leakage, Rework Leakage, Access Friction and Experience Driver Health.
- XP has been relabelled in the UI as Level Progress. Arena Points remain the spendable currency.
- Repeat-contact and rework dollar values are modeled and should be calibrated with client data before operational use.


## Client Outcome Intelligence Layer

This version makes Client Outcomes the focal point for TL and Manager views. The model follows: Client Outcome -> Outcome Score -> Driver Tree -> Root Cause Theme -> Value Bridge -> Recommended Intervention. Agents see controllable behaviors and Level Progress; TLs see team/process coaching indicators; Managers see account-level revenue, penalty exposure, modeled leakage, and intervention actions. ASA is kept as a TL/Manager access/capacity signal, not an Agent scorecard KPI.

## Final outcome-intelligence acceptance notes

This version is organized around three role experiences:

- **Agent:** controllable call-centre behaviors, Arena Points, Level Progress, missions, challenges and scorecard. ASA and abandonment are not agent-owned KPIs. Agent screens do not show client-dollar savings.
- **Team Lead:** team-scoped performance, SLA/KPI Trends, Client Outcomes, Driving Client Outcomes, Team Pulse, coaching/challenge actions and low team-scoped commercial exposure.
- **Manager:** account-level Client Outcomes, SLA/KPI Trends, Driving Client Outcomes, Revenue & Commercial, What-If / Action Planner, team comparison and adoption.

Revenue model: **Revenue MTD = Billable Calls × Rate per Call**. Penalty exposure is kept below 5% of modeled revenue. TL exposure is intentionally much lower than manager/account exposure for demo credibility.

## Excel-first dataset workflow

The prototype now includes an editable Excel source of truth so the seed data can be changed without hand-editing `data.js`.

### Files

- `Performance_Arena_Dataset.xlsx` — editable workbook with a README sheet plus one sheet per entity in `data.js`.
- `export_to_excel.py` — reads `data.js` and creates/refreshes `Performance_Arena_Dataset.xlsx`.
- `export_to_json.py` — reads the workbook and regenerates `data.js`.
- `validate_data.py` — validates key referential integrity and commercial scoping rules.

### Recommended edit workflow

1. Back up the working folder.
2. Run:

   ```cmd
   python export_to_excel.py
   ```

3. Open `Performance_Arena_Dataset.xlsx` in Excel.
4. Edit business-safe values such as targets, actuals, rewards, dates, statuses, commercial assumptions, and descriptions.
5. Avoid changing IDs unless you also update every linked sheet.
6. Save the workbook.
7. Regenerate the app data:

   ```cmd
   python export_to_json.py
   python validate_data.py
   node test_prototype.js
   ```

8. Expected final result:

   ```text
   140 PASS / 0 FAIL
   ```

9. After changing `data.js`, reset browser state before retesting:

   ```js
   localStorage.removeItem('arena_state_v7'); location.reload();
   ```

### Workbook guardrails

- Team Lead commercial rows must stay team-scoped.
- Manager commercial rows must stay account-scoped.
- Manager account penalty must stay higher than each individual TL team penalty.
- Overall account penalty should remain at or below 5% of modeled revenue for demo credibility.
- Keep active UI and data in USD notation only (`$`, `K`, `M`). Do not use INR, `₹`, lakh, crore, or Indian currency shorthand.
- Agent-visible scorecards should not include ASA, Call Abandonment, Claims Accuracy, Compliance Score, Claims Back Office metrics, claims TAT, or Productivity as a label.
