## 2. Runtime architecture

Performance Arena is a **static single-page app**.

Core runtime flow:

```text
index.html
  → loads styles.css
  → loads Tailwind CDN and Lucide CDN
  → loads data.js
  → loads app-core.js
  → loads app-views-agent.js
  → loads app-views-lead-mgr.js
  → loads app-modals.js
  → calls window.Arena.boot()
```

### 2.1 Static single-page app

- No backend server is required beyond static file hosting.
- No database is used at runtime.
- The app can run from a local Python static server or Netlify/static hosting.

### 2.2 Frameworks and libraries

The prototype is built in:

- HTML
- CSS
- Vanilla JavaScript
- Tailwind CSS CDN for utility styling
- Lucide icons CDN for icons

There is no React/Vue/Angular build step.

### 2.3 Data loading

`data.js` assigns the full seed dataset to:

```js
window.SEED_DATA = { ... }
```

`app-core.js` reads `window.SEED_DATA`, normalizes it into runtime state and provides helper functions for views and mutators.

### 2.4 State storage

The prototype stores simulated interactions in browser `localStorage`.

Known localStorage key:

```text
arena_state_v6
```

This stores local mutations such as:

- Accepted/declined challenges
- Challenge validation state
- Reward redemptions
- Training completions
- Mission progress
- Role/user/navigation state where implemented

Reset command:

```js
localStorage.removeItem('arena_state_v6'); location.reload();
```

### 2.5 Role switching

There is no real login/authentication. Role switching is a demo affordance.

Roles:

- Agent
- Team Lead
- Manager

The active role and active user determine which navigation and views are rendered.

### 2.6 Modals and dispatcher

User interactions are driven by data attributes and a central dispatcher in the core app. Buttons and forms use `data-action`, `data-nav`, or role-specific action IDs. `app-modals.js` renders modals and binds submit/cancel behavior.

## 3. Main modules

### 3.1 `app-core.js`

Responsibilities:

- Initialize app state from `window.SEED_DATA`.
- Merge seed data with localStorage mutation state.
- Provide entity lookup helpers.
- Provide role and user helpers.
- Provide KPI, score, RAG and commercial helpers.
- Manage navigation and role switching.
- Dispatch button/form actions.
- Mutate runtime state for missions, challenges, rewards, training and PKT.
- Persist local state back to localStorage.
- Reset prototype state.

Typical helper categories:

| Category | Examples |
|---|---|
| Users / roles | user lookup, active user, team membership, role switching. |
| KPI / score | current KPI rows, scorecards, RAG counts, trends. |
| Commercial | exposure, revenue, penalty, scope filtering. |
| Challenges | create, accept, reject, submit win, TL validation. |
| Missions | accept, progress, complete. |
| Rewards | redeem, approve, reject. |
| Learning | acknowledge broadcast, complete training, PKT attempt. |

### 3.2 `app-views-agent.js`

Agent view file. It renders:

- Agent Home
- Scorecard
- Missions
- Challenges
- Leaderboard
- Arena Store
- Training / Broadcast / PKT
- Profile / reset state

Key design logic:

- Agent-facing metrics exclude ASA.
- Agent-facing metrics exclude claims/back-office metrics.
- Agent views do not show modeled client-dollar savings.
- Challenges use Pts and require TL validation for claimed wins.
- XP is presented as Level Progress in demo-facing UI.

### 3.3 `app-views-lead-mgr.js`

Team Lead and Manager view file. It renders:

#### Team Lead pages

- Client Outcomes
- SLA/KPI Trends
- Client Metric RCA
- Team Pulse
- Coach Console
- Team view
- Missions / Challenges
- Commercial
- Training Console
- Coaching
- Recognition / rewards where applicable

#### Manager pages

- Client Outcomes
- SLA/KPI Trends
- Client Metric RCA
- Revenue & Commercial
- What-If / Action Planner
- Team Comparison
- Adoption
- Command Center / legacy manager support pages where applicable

Key design logic:

- TL pages are team-scoped.
- Manager pages are account-scoped.
- TL commercial exposure is significantly lower than account exposure.
- Manager revenue is modeled as billable calls multiplied by rate per call.
- Client outcome pages include outcome score, RAG, drivers, root-cause themes and recommended actions.

### 3.4 `app-modals.js`

Modal file. It contains forms and modal interactions for:

- Challenge creation
- Mission/action creation
- Training/PKT workflows
- Reward redemption/approval where modal-based
- Other CTA flows triggered from pages

Modal design requirements:

- Should be scrollable on mobile.
- Submit/cancel buttons must remain reachable.
- Role context must be preserved after TL/Manager actions.

## 5. State model

### 5.1 Initial seed data

Seed data is loaded from `data.js` into `window.SEED_DATA`.

Main entity groups include:

- Users
- Teams
- Processes
- KPI_Master
- Performance_Data
- Daily_Agent_Score
- Agent_Current
- Leaderboard
- Points_Ledger
- XP_Ledger / Level Progress data
- Missions and Mission_Assignments
- Challenges, Challenge_Participants, Challenge_Results
- Badges and Agent_Badges
- Rewards and Reward_Redemptions
- Communications and Communication_Status
- Learning_Modules, Learning_Assignments and completion status
- PKT assessments/questions/attempts
- SLA commercial rules and slabs
- Commercial exposure and verification
- What-if scenarios
- Coaching and Recognition
- TL/Manager verification

### 5.2 Mutated state

Runtime mutations are written to localStorage.

Examples:

- Mission accepted/progressed/completed.
- Challenge sent/accepted/declined/submitted for validation/validated.
- Reward redeemed/approved/rejected.
- Broadcast acknowledged.
- Training completed.
- PKT attempted.
- Recognition/coaching updates.

### 5.3 Reset behavior

Reset removes localStorage mutations and reloads the app.

```js
localStorage.removeItem('arena_state_v6'); location.reload();
```

### 5.4 Challenge lifecycle

Intended challenge flow:

```text
Agent creates challenge
→ Opponent receives challenge
→ Opponent accepts or declines
→ Active challenge
→ Agent submits win for TL validation
→ TL validates or rejects
→ Points awarded only after validation
→ Challenge moves to completed/settled state
```

Rules:

- Challenges use Pts as reward currency.
- Avoid gambling language such as “stake.”
- Win is not immediately awarded without TL validation.
- TL/Manager actions must preserve role context.

### 5.5 Mission lifecycle

Typical mission flow:

```text
Mission available
→ Agent accepts
→ Progress logged
→ Completed
→ Points / Level Progress awarded
```

### 5.6 Reward lifecycle

Typical reward flow:

```text
Agent redeems reward
→ Instant reward fulfilled or TL approval required
→ TL approves/rejects
→ Wallet updated if approved
```

### 5.7 Training/PKT lifecycle

Typical learning flow:

```text
Broadcast / Training / PKT assigned
→ Agent acknowledges or starts
→ Completion / attempt recorded
→ Points / Level Progress awarded if applicable
```

## 6. Known technical limitations

- Static prototype, no backend.
- No real authentication/authorization.
- Role switching is a UI affordance only.
- State is browser-specific localStorage.
- CDN dependencies exist for Tailwind and Lucide.
- Synthetic data only.
- Tests are code/render-level tests, not full browser automation.
- A manual Netlify and mobile walkthrough is still required before any live demo.
- Commercial/revenue/RCA logic is modeled and should be calibrated with client data before operational use.
